### 2.2 日常循环

```
你的一次迭代：

  ┌─ 你有一个想法 ──────────────────────────────┐
  │  例如："给声纳加回声衰减"                      │
  └──────────────┬──────────────────────────────┘
                 │
     ┌───────────▼───────────┐
     │ 重大功能？还是小改动？  │
     └───┬──────────────┬────┘
         │              │
     重大功能        小改动/bug
         │              │
         ▼              ▼
   逐步确认模式     一口气模式
   (见 2.3)        (见 2.4)
```

### 2.3 重大功能流程（逐步确认）

```
Step 1 — 讨论（你 + 任意 Instance）
   你: "我想加 [功能]，你觉得怎么实现？"
   AI 给出方案，你拍板
   ✅ 你确认方向

Step 2 — 文档先行（Instance A / CIO）
   你: "以 CIO 身份，为 [功能] 更新 GDD 和 TDD，写 ADR"
   CIO 输出文档 → git commit
   ✅ 你审阅文档，确认规范正确

Step 3 — 实现（Instance B / Engineer）
   你: "以 Engineer 身份，按 docs/ 里的规范实现 [功能]"
   Engineer 写代码 → git commit
   ✅ 你验收：能编译？能运行？符合预期？

Step 4 — 同步
   你告诉 CIO: "git log 看最新 commit，检查代码是否符合文档"
   CIO 更新 API 文档 → git commit
   ✅ 文档和代码一致
```

### 2.4 小改动流程（一口气）

```
直接告诉 Engineer:
   "修复 [问题] / 实现 [小功能]，完成后 commit"

完成后告诉 CIO:
   "git log 看最新改动，更新相关文档"

你最后验收
```

## 7. 黄金法则

1. **先找总设计师** — 有想法先和 Architect 讨论，他会告诉你启动谁
2. **小步提交** — 让 Agent 频繁 commit，别攒大改动
3. **你不写代码也不写文档** — 你的时间花在创意和验收上
4. **有疑问就问** — 对任意 Instance 说 "解释一下 [X]"
5. **文档 >= 代码** — 如果文档没更新，这个功能就没完成
6. **信任流程** — 总设计师调度，你执行，Agent 产出，你验收
